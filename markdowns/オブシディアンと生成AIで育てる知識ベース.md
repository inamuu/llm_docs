どちらの投稿も短いですが、含んでいる示唆は大きいものです。

ただのメモアプリの話ではありません。

自分の知識をどうためるか。

この本では、まず二つの投稿を日本語で訳します。

# 2. カーパシーの投稿を訳す
## 生成AIで個人用の知識ベースをつくる
最近かなり便利だと感じているのは、関心のある研究テーマごとに、生成AIを使って個人用の知識ベースをつくることだ。

このやり方では、最近自分が使っているトークンの多くが、コードをいじることよりも、知識をいじることに使われている。

その知識は、Markdownや画像として保存されている。

最新の生成AIは、こういう作業がかなり得意だ。

## データの取り込み
まず、記事、論文、リポジトリ、データセット、画像などの元資料を `raw/` ディレクトリに集める。

そのあと生成AIを使って、少しずつ wiki を「コンパイル」していく。

この wiki は、ディレクトリ構造の中に並ぶ Markdown ファイルの集まりにすぎない。

wiki には、`raw/` にある資料の要約、相互参照、概念ごとの分類、それぞれの概念についての記事、そしてそれらを結ぶリンクが含まれる。

ウェブ記事を Markdown 化するときには Obsidian Web Clipper を使うのが気に入っている。

関連画像もローカルに保存できるようにしておくと、生成AIが参照しやすくなる。

## IDE としての Obsidian
自分は Obsidian を、フロントエンド側の IDE のように使っている。

そこでは元資料、コンパイル済みの wiki、派生した可視化をまとめて見られる。

大事なのは、wiki のデータは生成AIが書き、維持していることだ。

自分はその中身を直接いじることはめったにない。

別の表示方法を試すために、いくつかの Obsidian プラグインも使ってきた。

たとえば Marp でスライド表示をするような使い方だ。

## 問い合わせと調査
面白くなるのは、wiki が十分な大きさになってからだ。

たとえば百本ほどの記事、四十万語くらいの規模になると、生成AIエージェントに対して、かなり複雑な質問を投げられる。

するとエージェントは wiki を調べ、必要な情報を読み、答えを組み立ててくれる。

最初は本格的な RAG が必要だと思っていた。

だが、このくらいの比較的小さな規模なら、生成AIが索引ファイルや各文書の短い要約を自動で保守しながら、重要な関連情報をかなりうまく読めている。

## 出力の形
答えをテキストやターミナルだけで受け取るより、自分は Markdown ファイル、Marp 形式のスライド、Matplotlib の画像として出力してもらうのが好きだ。

それらをまた Obsidian で見る。

質問に応じて、ほかにもいろいろな可視化の形式が考えられる。

しかも、その出力をふたたび wiki に収めることが多い。

そうすると、自分の探索や質問が、そのまま知識ベースを育てることになる。

## 点検と整備
wiki に対して、生成AIによる健康診断のようなチェックも走らせている。

たとえば、矛盾したデータを見つけたり、ウェブ検索を使って欠けた情報を補ったり、新しい記事候補になる面白い関連を見つけたりする。

そうやって wiki を少しずつ掃除し、全体の整合性を高めていく。

生成AIは、次に何を調べるべきかを提案するのも上手い。

## 追加の道具
データ処理用の追加ツールも作るようになってきた。

たとえば、wiki の上に小さくて素朴な検索エンジンを自作した。

自分でウェブ画面から使うこともあるが、もっと多いのは、生成AIにCLIツールとして渡し、大きな問い合わせに使わせるやり方だ。

## さらに先の話
リポジトリが大きくなると、合成データの生成やファインチューニングも考えたくなる。

つまり、知識をコンテキストウィンドウだけで扱うのではなく、モデルの重みにも覚えさせたくなる。

## まとめの意訳
要するに、いくつかの情報源から元データを集め、それを生成AIが Markdown の wiki に編成し、さらに各種のCLIツールを使いながら、質問応答と知識ベースの改善を続けるということだ。

その全体を Obsidian から見渡せるようにする。

wiki を人が手で書く場面は少ない。

そこは生成AIの作業場になる。

そして、このやり方には、寄せ集めのスクリプトではなく、一つの優れた製品になりうる余地がある。

# 3. kepano の投稿を訳す
## なぜこのやり方がよいのか
このやり方がよいと思うのは、エージェントが生成した内容で、自分の主たる vault が汚染される危険を減らせるからだ。

エージェントにも遊び場が必要だ。

カーパシーの Obsidian の使い方は、その汚染リスクを下げる方法としてうまい。

自分の personal vault はきれいに保ち、エージェント向けには別の、多少散らかった vault を用意すべきだ。

## 自分の vault を高信号に保つ
自分の personal vault には、情報密度が高く、雑音が少ない状態であってほしい。

しかも、そこにある内容の出どころはわかっていてほしい。

自分が書いたものと、エージェントが作ったものを分けておけば、出典のあいまいな考えや文章で主たる vault が汚染されるのを防げる。

この二つが混ざりすぎると、Obsidian はだんだん使いにくくなる。

検索、Bases、クイックスイッチャー、バックリンク、グラフといった機能が、**自分の知識**を表すものではなくなってしまうからだ。

## 価値が出たものだけを持ち込む
まずはエージェント側の作業フローで有用な成果物をつくる。

そのうえで、本当に残す価値があるものだけを primary vault に持ち込むべきだ。

## 低放射線鋼の比喩
1940年代には、核爆弾の爆発で地球上の鉄鋼の多くが放射性降下物にさらされた。

そのため、放射線量の低い戦前の鋼材を、海底の沈没船から回収する必要が生じた。

2020年代には、ChatGPT によって合成テキストが大量に広がった。

その結果、今度は生成AI以前の情報源を分けて保全したくなる。

こうした比喩で、彼は「AI以前の資料を隔離して守る必要」を表現している。

# 4. 二つの投稿が伝えていること
二つの投稿は、同じ方向を向いています。

カーパシーは、生成AIを知識の編集者として使う発想を出しました。

kepano は、その発想を受け止めつつ、**どこまでをAIの領域にし、どこからを自分の領域にするか**を明確にしました。

この組み合わせが重要です。

生成AIは、集める、要約する、つなぐ、再構成する作業が得意です。

一方で、出典の混線、誤りの増幅、似た文章の大量発生も起こしやすい。

だから、生成AIの生産性だけを見るのでは足りません。

どこに保存し、どう昇格させ、何を自分の正式な知識と見なすかまで設計する必要があります。

Obsidian は、この設計に向いています。

理由は単純です。

中身がローカルの Markdown ファイルだからです。

つまり、アプリの中に閉じ込められたデータではなく、あとから人でもAIでも処理しやすい「普通のファイル」として知識を持てます。

# 5. Obsidian を使ったことがない人のための基本
## vault はただのフォルダ
Obsidian でいう vault は、特別なデータベースではありません。

実体は、Markdown ファイルや画像が入ったローカルのフォルダです。

この考え方がかなり大事です。

あなたの知識は、ある会社のサーバーの中にあるのではなく、自分の手元のファイルとして残ります。

検索もしやすく、コピーもしやすく、他のツールでも扱えます。

## ノート同士をリンクで結ぶ
Obsidian の強みは、メモを書くこと自体より、メモ同士をつなぐことにあります。

一つひとつのノートは小さくても、リンクで関係を明示していくと、あとからたどりやすくなります。

カーパシーの話で言えば、元資料、概念ノート、要約、質問への回答を、別々の Markdown ファイルとして持ち、それらをリンクでつなぐ形です。

## 生成AIとの相性がよい
Markdown は、生成AIが最も扱いやすい形式の一つです。

画像も同じフォルダに置けます。

つまり、Obsidian は「人間が読む前面」と「生成AIが加工する背面」が同じ土台の上にある道具だと考えるとわかりやすくなります。

# 6. 2026年4月時点での、うまい使い方
## personal vault と agent vault を分ける
いちばん大事なのはここです。

自分で書くノートの vault と、生成AIに触らせる vault を分けます。

前者は、あなたの考え、仕事の記録、確度の高いメモを置く場所です。

後者は、記事の取り込み、荒い要約、比較表の下書き、仮説メモ、画像整理、調査ログの置き場です。

この二層構造にすると、AIの試行錯誤を活かしながら、自分の思考の輪郭も守れます。

## agent vault の中身は四つでよい
最初は複雑にしない方が続きます。

`raw` に元資料を置く。

`wiki` に概念ノートと要約を置く。

`outputs` にレポートやスライドを置く。

`logs` に実行メモや確認事項を置く。

この四つで十分です。

カーパシーの発想は高度に見えますが、骨格はとても単純です。

## AIの生成物は昇格制にする
agent vault で生まれたノートを、すぐに personal vault へ入れないことです。

一度読んで、役に立つか、出典が追えるか、自分の言葉で説明できるかを見ます。

その三つを満たしたものだけを、手で持ち込みます。

この一手間が、kepano のいう「汚染防止」にあたります。

## 最初から全部自動化しない
よくある失敗は、最初から巨大な仕組みを作ることです。

最初の一か月は、ウェブ記事の保存、要約、概念ノート化、質問への回答出力までで十分です。

検索エンジンや複雑な自動分類は、情報がたまってからで遅くありません。

# 7. 役に立つ機能を順番に見る
## Web Clipper
2026年4月時点の Obsidian 公式ヘルプでは、Web Clipper は、ページをハイライトし、ウェブの内容を vault に保存する無料のブラウザ拡張として案内されています。

ローカル保存が前提で、コードも公開されています。

この性質は大きな安心材料です。

生成AI用の材料集めでは、あとで引用元を見返せることが重要だからです。

まずは記事の保存をここから始めるのが自然です。

## Backlinks
公式ヘルプによると、Backlinks は、今見ているノートを他のどのノートが参照しているかを見せる機能です。

しかも、明示的にリンクしたノートだけでなく、名前が出ているだけの未リンク言及も拾えます。

これは知識ベースの保守に向いています。

ある概念ノートが、どこで使われ、どこで放置されているかが見えるからです。

## Graph view
Graph view は、ノート同士の関係を図として見せる機能です。

大きな vault では飾りに見えることもありますが、最初のうちはかなり有効です。

リンクの偏りや、孤立したノート、集まりすぎたハブが見えるからです。

知識ベースが「ただのファイル置き場」になっていないかを、ざっと点検できます。

## Canvas
Canvas は、ノート、画像、PDF、ウェブページなどを二次元上に並べて関係を線で結べる機能です。

公式ヘルプでは、`.canvas` という open な JSON 形式で保存されると説明されています。

この点がよいところです。

見た目は視覚的でも、裏ではファイルです。

テーマの整理、比較、講義メモ、構想づくりに向いています。

## Properties
Properties は、ノートに構造化された属性を持たせる機能です。

公式ヘルプでは、テキスト、リスト、数値、チェックボックス、日付、日時、タグなどを扱えるとされています。

たとえば、記事ノートに「出典」「日付」「信頼度」「要確認」といった属性をつけておくと、あとでAIにも人にも扱いやすくなります。

これは小さな工夫ですが、あとで効きます。

## Bases
Bases は、ノートをデータベースのように見せる機能です。

公式ヘルプでは、ノートと Properties を使って、並べ替え、絞り込み、複数ビューでの表示ができると説明されています。

読む記事一覧、調査中のテーマ、未確認のメモ、昇格候補のノートなどを見渡すのに向いています。

Obsidian を単なるメモ置き場から、軽い研究環境へ押し上げるのがこの機能です。

# 8. low-background steel とは何か
## まず用語の意味
英語版Wikipediaの「Low-background steel」は、これを "steel produced prior to the detonation of the first nuclear bombs" と説明しています。

つまり、最初の核爆弾が爆発する前につくられた鋼です。

なぜそんな鋼が特別扱いされるのかというと、核実験以後の大気には人工的な放射性物質が広がり、その空気を使ってつくられた鉄鋼には、ごく弱い放射能の痕跡が入りうるからです。

Wikipedia では、ベッセマー法や転炉法のように大気中の気体を使う製鋼工程は、空気中の粒子による汚染を受けうると説明しています。

そのため、非常に敏感な測定機器では、戦前の鋼材が重宝されてきました。

## なぜ沈没船から回収するのか
この種の鋼は、古い船や沈没船から得られることがあります。

特に戦前や戦間期の鋼材は、核実験以前の大気環境でつくられているため、検出器の遮蔽材として価値を持ちました。

Wikipedia は、スカパ・フローで自沈した艦船群が、その供給源の一つだったと述べています。

また、古代ローマの鉛のように、ほかの低バックグラウンド素材も高感度の実験で使われてきました。

ここで重要なのは、素材そのものの値段より、**雑音の少なさ**に価値があるという点です。

## いまでは完全に必要なのか
英語版Wikipediaによれば、1963年の部分的核実験停止条約以後、人為的な放射線の影響は下がり、2008年時点では多くの用途で特別な低バックグラウンド鋼は必須ではなくなったとされています。

ただし、きわめて高感度な機器では、今も需要が残ります。

ここは大事な点です。

比喩としては強烈ですが、現実には「すべてが完全に使えなくなった」という話ではありません。

雑音が少ない素材が、特定の用途で今も意味を持つという理解が近いでしょう。

## なぜAI時代の比喩として効くのか
kepano がこの話を持ち出した理由も、そこにあります。

生成AIの時代になると、文章や画像の全体量は一気に増えます。

しかし、量が増えることと、信頼できることは同じではありません。

しかも、一度大量の合成テキストがネットに広がると、あとから「人が書いた一次資料だけを見たい」と思っても、見分けがつきにくくなります。

だから彼は、低放射線鋼の比喩を使って、**低雑音の情報源を守る必要**を示したのです。

この比喩は少し大げさですが、感覚としてはかなり伝わります。

自分の primary vault を、出所不明の要約や連鎖的なAI文章で埋めない方がよい、という警告として読むとわかりやすいでしょう。

# 9. あなたならどう使うとよいか
## 最初の結論
あなたは、いきなり Obsidian を「第二の脳」として始める必要はありません。

まずは、**読む資料をためる場所**と、**自分が残したい考えを書く場所**を分けるだけで十分です。

この分け方が、あとから効いてきます。

## おすすめの始め方
1. personal vault を一つ作る。
2. agent vault を別に一つ作る。
3. 読んだ記事や資料は agent vault に入れる。
4. AIに要約、分類、関連づけをさせる。
5. 本当に残したいものだけを personal vault に移す。

この順番なら、生成AIの便利さを使いながら、自分の思考の中心も守れます。

## personal vault に入れるもの
personal vault に入れるのは、日記でも雑記でもなく、あとで自分が何度も参照するものです。

たとえば、仕事で使う考え方、判断基準、読書メモの要点、長く残したい学びです。

ここは量より質です。

検索したときに、自分の言葉で書かれた確かなものが出てくる状態を目指します。

## agent vault に入れるもの
agent vault には、記事クリップ、論文メモ、画像、荒い要約、未確認の仮説、比較メモ、調査ログを置きます。

多少散らかっていても問題ありません。

ここは整った書斎ではなく、作業場です。

AIが下書きを量産し、関係をつなぎ、欠けを見つける場所だと割り切ります。

## 持ち込む基準を決める
agent vault から personal vault に持ち込むときは、最低でも三つを確認します。

出典が追えるか。

自分で読んで納得できるか。

自分の言葉で短く言い直せるか。

この三つを通らないものは、たとえ便利でも昇格させない方が安全です。

## 仕事にどう効くか
この運用は、読書管理だけに使うものではありません。

業界調査、競合比較、顧客理解、制度変更の追跡、技術キャッチアップにも向いています。

特に会社員にとって価値があるのは、情報を読んで終わりにせず、何度でも再利用できる形にする点です。

生成AIはその整形係としてかなり役に立ちます。

# 10. 最後に
カーパシーの提案は、生成AIを知識編集の実務に入れる発想でした。

kepano の補足は、そこに必要な境界線を与えました。

つまり、AIを使うことより、**AIをどこで使うか**の方が大事だということです。

Obsidian をこれから始めるなら、まずはローカルの Markdown を土台にし、自分の vault と AI の vault を分けてください。

Web Clipper で集める。

Properties と Bases で整理する。

Backlinks と Graph でつながりを見る。

Canvas で構想を視覚化する。

そして、価値が確認できたものだけを、自分の本棚へ戻す。

この流れなら、生成AIを使っても、自分の知識の輪郭を見失いにくくなります。
